The probe's payload (`results.md`) dictates the tier order: **decide at the cheapest tier
that can decide.** Anything a shell/test gate settles (MC-02/06/08/12, JG-09) never reaches
the judge; the judge is spent only on the subtle cases where it scored 100% and stayed
stable.

## Design + recommendation

### 1. The goal/acceptance object (shared, loop + stack scope)

New `lopi-core` type â€” one schema, used at both scopes (see `cross-cutting.md`):

```rust
/// A machine-checkable success condition for a loop or a stack.
pub struct Acceptance {
    /// All must pass for the goal to be met (AND semantics; order = tier order).
    pub checks: Vec<AcceptanceCheck>,
}
pub struct AcceptanceCheck {
    pub tier: EvalTier,              // ExecutionOk | ShellTest | Judge | Suite
    pub spec: CheckSpec,             // tier-specific payload
    pub weight: f32,                 // for the scalar score (A3 ratchet reads this)
    pub required: bool,              // hard gate vs soft signal
}
pub enum EvalTier { ExecutionOk, ShellTest, Judge, Suite }
pub enum CheckSpec {
    ExecutionOk,                                   // builds/runs clean
    Shell { cmd: String },                         // exit 0 = pass  (reuses run_guard_command)
    Judge { rubric: Rubric, metric: Option<MetricGate> }, // separate-model verdict
    Suite { name: String },                        // KCQF etc.
}
pub struct MetricGate { pub name: String, pub op: Op, pub threshold: f64 } // e.g. coverage >= 0.8
```

`EvalTier` maps 1:1 onto the UI's existing `EvalTier` string union (`stack.ts:21`) â€” so the
inert `EvalRef` tags become the *authoring surface* for `AcceptanceCheck`s. This is the
bridge that makes the eval UI finally execute.

### 2. The evaluator interface (one trait, all tiers pluggable â€” see cross-cutting)

```rust
#[async_trait] pub trait Evaluator {
    async fn evaluate(&self, ctx: &EvalContext) -> EvalOutcome;
}
```
- `EvalContext` = { goal, full diff (NOT a 6 KB prefix â€” caveat #1 from the probe), test
  output, repo path, metric readings }.
- Four impls: `ExecutionOkEval`, `ShellTestEval` (wraps `run_guard_command`), `JudgeEval`
  (wraps `VerifierAgent::verify` â€” **reuse, don't rebuild**), `SuiteEval` (wraps KCQF).
- A `TieredEvaluator` runs `Acceptance.checks` in tier order, short-circuiting on the first
  *required* failure at a cheap tier before paying for the judge.

### 3. The verdict â€” one result, three consumers (see cross-cutting)

```rust
pub struct EvalOutcome {
    pub verdict: Verdict,            // Pass | Fail | Error
    pub score: f32,                  // weighted scalar in [0,1] â€” A3 ratchet reads this
    pub per_check: Vec<CheckResult>, // tier, pass/fail, gaps, fix_hints, confidence
    pub critique: Vec<String>,       // flattened fix_hints â†’ A2 reflection reads this
}
```
Persist it next to attempts so A3's ratchet and B1's stack goal can query it (new
`eval_outcomes` table keyed by task_id+attempt, mirroring `verifier_verdicts`).

### 4. Where it hooks into the run loop

Replace the bare `score.passed()` decision at `run_loop.rs:363-364` with a
`TieredEvaluator::evaluate()` call whose `Acceptance` comes from the task (falling back to
"tests pass + lint clean" â€” i.e. today's `score.passed()` â€” when no acceptance is set, so
behavior is unchanged for existing tasks). The judge tier here **subsumes** the finalize
verifier for tasks that declare a judge check; keep the finalize verifier as the L3/L4
autonomy gate for tasks that don't (backward-compatible).

### 5. Close the fail-open hole

`JudgeEval` on API/parse error must return `Verdict::Error`, and a *required* judge check in
`Error` state must **block** finalize (configurable: fail-closed for L3/L4, fail-open only
if the operator opts in). This fixes the `verifier_runner.rs:54-57` hole for the verified-PR
guarantee.

**Recommendation:** build the `Acceptance` object + `Evaluator` trait + `TieredEvaluator` +
`eval_outcomes` persistence, reusing `VerifierAgent`/`run_guard_command`/`Scorer` as tier
impls. Wire the UI `EvalRef` tags â†’ `Acceptance` in `cardToTaskPayload`. Do **not** rebuild
the judge. Size: **M** (the hard parts â€” the judge, shell exec, persistence pattern â€” exist).

## Pre-registered BUILD kill-test

> **Pre-register before building (do not move after seeing results):**
> A loop given an `Acceptance` with a failing check scores **FAIL**; the same loop with a
> passing check scores **PASS**; the judge-tier verdict is produced by a model **â‰ ** the
> author model; and the verdict is **stable across 3 reruns** on a fixed fixture. Plus: an
> `Acceptance` with a *required* judge check whose API errors scores **Error â†’ blocked**
> (fail-closed), not silently passed. Threshold: 100% correct on a 6-case fixture (2 pass, 2
> fail, 1 wrong-model-rejected, 1 error-blocked), 0 flips across 3 reruns.

## Biggest risk

**Input completeness, not judgment** (probe caveat #1). If `JudgeEval` is handed a truncated
diff or a summarized metric, it grades blind and the whole ratchet inherits that blindness.
Mitigation: pass the full file-scoped diff and raw metric readings into `EvalContext`;
prefer a machine-checkable `MetricGate`/`Shell` check over a judge check whenever the
criterion can be made objective. Secondary risk: scope creep turning this into a
half-built everything-evaluator â€” hold tiers to the 4 defined; `Suite` can be a thin KCQF
wrapper for v1.

## Open questions

   judge-declaring tasks? (Recommendation: subsume, with finalize verifier as the fallback
   autonomy gate â€” but confirm we don't want both for defense-in-depth at L4.)
2. `Acceptance` semantics: strictly AND across checks, or allow weighted soft-pass for
   non-`required` checks feeding only the A3 score? (Leaning: required = AND gate,
   non-required = score-only signal.)
3. Full diff vs file-scoped diff into `EvalContext` â€” where is the token-cost line for large
   diffs, and do we chunk-and-map the judge over files?
4. Do we let a `MetricGate` read arbitrary tool output (coverage %, bench number), and if so
   how is that reading produced and trusted (it is itself gameable â€” see JG-06/JG-11)?
