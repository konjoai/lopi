# R-A3 · Progress-gating + termination + budget enforcement

**Reframe:** the no-progress stall guard already exists and works (it halts a stuck loop
today). A3 is: **add the beats-best ratchet** (reject regressions), **persist the score
history so the gate survives and the orchestrator can read it**, connect the single-run
score to the **30-run statistical rigor** lopi already owns, and **meter budget in the
loop** so it actually caps.

---

## Current state (from §3, §4, §5)

- **No-progress detection exists and is live.** `update_no_progress_streak` increments when
  the weighted score fails to improve beyond EPSILON, resets on real improvement
  (`finalize.rs:238-251`); the loop halts with `NoProgressStall` once the streak hits
  `no_progress_limit` (`run_loop.rs:448-464`; limit from `.lopi/loop.toml`,
  default 3, `loop_config.rs:305`, `finalize.rs:202`). **But it is in-memory only** — nothing
  persists it where the orchestrator or a stack can read it.
- **No ratchet.** The loop accepts an attempt when `score.passed()` (`run_loop.rs:364`); it
  never *rejects an iteration for regressing* against the best-so-far. `best_weighted` is
  tracked (`run_loop.rs:65`) only to feed the stall streak, not to gate acceptance.
- **Score history: raw rows, no query.** `attempts.score_*` per `attempt_num`
  (`schema.sql:10-21`); `run_attempts(task_id)` returns the oldest-first series
  (`run_trace.rs:88-99`). **No "improving/stalled/regressed over last N" API** (§5).
- **30-run rigor lives in benchmarks, not the live loop.** The paired-Wilcoxon/CoV machinery
  (`benchmarks/`) proves "did it beat best" statistically but is offline; a single live run
  emits one `Score`, no distribution.
- **Budget is hidden and unenforced.** `LoopConfig.budget_tokens` exists (`loop_config.rs:311`,
  `0` = inherit) and `cli_budget_usd` gets passed to the CLI (`run_loop.rs:90-92`), but there
  is no in-loop meter that stops the run when cumulative cost crosses a cap; the stack
  `budget` control is client-only and a `budget N` badge was *removed* because nothing
  enforced it (`StackConnector.svelte:6-14`). The turn cap (`max_turns`) is the only hard
  runaway stop (`run_loop.rs:100-109`).

## Prior art mapped onto lopi's assets

| Prior art | lopi already has | A3 adds |
|---|---|---|
| Hill-climb / accept-if-improves (ratchet) | `best_weighted` tracking + `Score::weighted` (`agent.rs:149`) | *reject* an attempt whose eval score < best (not just count stalls) |
| Plateau/early-stop detection | `update_no_progress_streak` + `no_progress_limit` (`finalize.rs:238`) | persist it; add a K-round window the orchestrator/stack can read |
| Paired significance testing (variance-aware acceptance) | 30-run Wilcoxon/CoV in `benchmarks/` | relate a single live score to the gate: when to demand a re-sample vs accept |
| Cost/budget ceilings (agentic runaway control) | `budget_tokens`, `cli_budget_usd`, `turn_metrics` cost rows (`schema.sql`) | an in-loop meter that sums `turn_metrics` cost and stops at the cap |

## Design + recommendation

### 1. Beats-best / ratchet

After A1's `EvalOutcome.score` for an attempt, compare against the persisted best-so-far for
this task (or stack). Accept the attempt into the "best" slot only if
`score > best + EPSILON`; a *regression* is rejected — the loop keeps the prior best branch
and either retries or (with no-progress) halts. This is `update_no_progress_streak`'s logic
promoted from "count stalls" to "gate acceptance," reading the **A1 eval score** instead of
the heuristic `weighted`.

**Variance coupling (the 30-run tie-in):** a single run's score is noisy (this is exactly
JG-06/JG-11 in the probe — one lucky run is not evidence). So the ratchet must not promote on
a single sample when the metric is variance-prone. Design: a check flagged `stochastic` in
its `MetricGate` triggers **re-sampling** — run the eval K times (small K, e.g. 3–5) and
gate on the *paired* comparison (median/Wilcoxon-lite) vs best, reusing the benchmarks'
statistical helpers. Deterministic checks (tests pass, shell exit) promote on one sample.
This is the 30-run rigor applied live, at the granularity the cost allows.

### 2. Persist score history + a progress query

New read API on `MemoryStore` over the `attempts`/`eval_outcomes` rows:
`score_trajectory(task_id) -> Vec<(attempt, score)>` and
`progress_state(task_id, window: K) -> {best, last, improving, stalled_rounds, regressed}`.
Built on `run_attempts` (`run_trace.rs:88`). This makes the stall/ratchet state **durable
and orchestrator-readable** (fixes §5 gap; B1 reuses it at stack scope).

### 3. No-progress termination (persisted)

Keep `no_progress_limit` semantics; persist the streak so a resumed/rescheduled run and the
stack sequencer can see "this loop has stalled K rounds → stop or escalate." Escalation hook:
on stall, optionally bump the self-prompt strategy (A2) or model tier before halting, rather
than a hard stop (config choice).

### 4. Budget enforcement (un-hide budget)

Add an in-loop **cost meter**: after each attempt, sum this run's `turn_metrics` cost
(`save_turn_metrics`, `store/mod.rs:290`; rows already carry per-turn cost/tokens) and
compare to the effective budget (`task` override → `LoopConfig.budget_tokens` → global).
Crossing the cap stops the loop with a terminal `BudgetExceeded` status — the same
early-terminal pattern as `NoProgressStall`/`TurnLimitExceeded` (`run_loop.rs:100-109`,
`:456-462`). This is what lets the UI budget control (`GuardrailsPopover` budget, the removed
badge) become real.

**Recommendation:** build in this order — (a) persist score history + progress query
(unblocks everything, small), (b) ratchet on deterministic checks (small), (c) budget meter
(small, independent), (d) stochastic re-sampling gate (medium, reuses benchmarks). Size:
**M** total, gated on A1 (need `EvalOutcome.score`), ideally after A2.

## Pre-registered BUILD kill-test

> **Pre-register (thresholds fixed before building):**
> 1. **Ratchet rejects regression:** a scripted run where attempt 2's eval score < attempt
>    1's → attempt 2 is *not* promoted to best and the loop retains attempt 1's result.
>    Binary.
> 2. **Plateau halts:** a run whose eval score is flat (within EPSILON) for
>    `no_progress_limit` rounds is stopped with `NoProgressStall` and **not** allowed to burn
>    the full `max_iterations`. Binary; assert iterations used < max.
> 3. **Budget stops:** a run with a low `budget_tokens` cap and a metered per-turn cost
>    stops with `BudgetExceeded` before `max_iterations`, at the first attempt that crosses
>    the cap. Binary; assert terminal status + attempt index.
> 4. **Variance guard (if stochastic gate shipped):** a `stochastic` metric that hits its
>    threshold on 1 of 5 samples but not on the paired median is **rejected**, not promoted
>    (the JG-06 case). Binary.

## Biggest risk

**Ratcheting on a noisy single score** — promoting a lucky run (JG-06/JG-11) is the exact
failure the 30-run rigor exists to stop, and a naive `score > best` ratchet re-introduces it
live. If we *don't* re-sample, the ratchet rewards variance; if we re-sample every check, cost
explodes. Mitigation: only stochastic checks re-sample, with small K, and deterministic
checks (the majority) promote on one sample. Get the `stochastic` flag right in A1's
`MetricGate`. Secondary risk: budget metering that reads stale/incomplete `turn_metrics`
undercounts and never fires — assert the meter sums *committed* rows and test the boundary.

## Open questions

1. What K for stochastic re-sampling in the live loop, and is a lightweight paired test
   (sign test / median-of-K) enough, or do we need the full Wilcoxon from benchmarks? (Cost
   vs rigor.)
2. On stall, hard-stop vs escalate-then-stop (bump strategy/model first)? Config default?
3. Budget unit: tokens, USD, or both? `budget_tokens` and `cli_budget_usd` both exist —
   reconcile into one effective meter.
4. Does the ratchet keep the best *branch* around (worktree cost) or just the best *score*
   for the gate? (Interacts with isolation mode.)
