# R-B1 · Goal-directed stacks

**Reframe:** turn the client-side stack sequencer from "advance the chain `loopTarget`
times" into "advance until the stack's **goal-evals pass** or termination fires," reusing
A1's `Acceptance` schema at stack scope and A3's progress/termination machinery. The UI
surface (the purple StackControlDock) already exists and already renders an inert "chain
acceptance" — B1 makes it real.

---

## Current state (from §7, §1, §3)

- The sequencer is **fixed-count**: `advance()` loops the whole chain until `repetition`
  reaches `loopTarget` (0 = infinite), a failure policy fires, or pause/drain
  (`stackRun.ts:131-223`, termination at `:149-160`). No goal-based stop.
- Launch snapshot freezes `order`, `loopTarget`, `onFail` (`stackRun.ts:246-250`).
- `StackConfig` = `loopCount, scheduled, cron, guardrails, evals, defaults`
  (`stack.ts:773-780`) — **no goal/acceptance field**; `evals` is inert "chain acceptance"
  copy (`stack.ts:831-839`).
- Stacks are **100% client-side** — no server-side stack/plan concept (`stackRun.ts:5-13`).
  So stack-goal evaluation has to decide: evaluate in the client (drive existing per-task
  eval results) or introduce a server-side stack runner.

## Prior art mapped onto lopi's assets

| Prior art | lopi already has | B1 adds |
|---|---|---|
| Goal-conditioned control loop (run until objective met) | per-task `Acceptance` (A1) + per-task ratchet/no-progress (A3) | the same, hoisted to *stack* scope |
| Orchestrator-workers with a shared objective | the stack chain + `cardToTaskPayload` fan-out | a stack-level goal the chain pursues |
| Outcome-based termination | A3 `progress_state` / no-progress | read it across the chain's task results |

## Design + recommendation

### 1. Where the stack goal lives

Add `goal?: Acceptance` (the **same A1 schema**, cross-cutting rule #1) to `StackConfig`
(`stack.ts:773-780`), initialized in `defaultStackConfig()` (`stack.ts:785-794`) next to
`loopCount`. The inert `evals: EvalRef[]` becomes the *authoring surface* for it — the
existing EvalsPopover ("chain acceptance") edits the stack goal's checks. Add it to the
launch snapshot (`stackRun.ts:246-250`) so a mid-run goal edit doesn't reshape an in-flight
run (matching the existing snapshot discipline).

### 2. How the chain termination changes

Today the only stop-check is `cursor >= order.length` → count (`stackRun.ts:149-160`). B1
adds a **goal check at the end of each full pass**: after a repetition completes, evaluate
the stack goal; if satisfied → finalize `done` (goal met); else if a stack-level no-progress
window (A3 `progress_state` across passes) shows no improvement for K passes → finalize
`stopped (no-progress)` with the reason recorded; else continue if `loopTarget` allows.
`loopTarget` becomes a **safety ceiling**, not the primary terminator — "pursue the goal, but
never more than N passes."

### 3. What evaluates the stack goal (recommendation)

Two options:
- **(A) Client-drives, reuses per-task evals.** The stack goal's checks are evaluated from
  the *already-computed* per-task `EvalOutcome`s of the chain's cards (A1 persists them;
  read via a status/eval source like the existing `waitForTerminal` status injection,
  `stackRun.ts:106-119`). A stack "metric ≥ X" goal reads the relevant card's metric result.
  No new server runner. **Recommended for v1** — smallest change, matches the client-only
  architecture, and the stack goal is usually "did the chain's key card hit its number."
- **(B) Server-side stack evaluator.** A backend endpoint runs the stack `Acceptance` fresh
  against repo state. More faithful for goals not reducible to a card's eval, but requires
  introducing the server-side stack concept the codebase deliberately avoids today. Defer to
  a later phase unless a real goal needs it.

**Recommendation:** ship **(A)**. `Acceptance` at stack scope, evaluated from persisted
per-task `EvalOutcome`s, terminating the sequencer on goal-met or stack-no-progress, with
`loopTarget` as the ceiling and the stop reason recorded in `StackRunState`. Size: **M**,
gated on A1 (schema + per-task eval results) and A3 (progress query). Depends on the
stack-control dock already shipped (Stack-1, PR #68).

## Pre-registered BUILD kill-test

> **Pre-register (before building):**
> A stack with goal "metric ≥ X" on its key card:
> 1. **loops past a single pass** when the metric is below X after pass 1 (does not stop at
>    `loopCount` if goal unmet and ceiling not hit), and **stops as `done`** on the pass
>    where the metric first reaches X;
> 2. **stops as `stopped(no-progress)`** — with the reason recorded in `StackRunState` — when
>    the metric plateaus below X for the K-pass window, instead of running to the ceiling;
> 3. **stops at the `loopTarget` ceiling** with reason `ceiling` if neither goal nor
>    no-progress fires first.
> All three assert the terminal phase + recorded `stopReason`. Mirror the existing
> `stackRun.test.ts` structure (`:305-369`).

## Biggest risk

**A stack goal that isn't reducible to the chain's per-task eval results** — option (A)
assumes the stack goal reads from cards' `EvalOutcome`s. A goal like "the whole repo's
benchmark ≥ X after the chain runs" may not correspond to any single card's eval, forcing
option (B)'s server-side evaluator and the server-side stack concept the codebase avoids.
Mitigation: scope v1 stack goals to *card-attributable* metrics (the common case: "loop 3's
coverage ≥ 0.9"); document that whole-repo/emergent goals wait for a server-side stack
runner. Secondary risk: variance at stack scope (a lucky pass "meets" the goal) — reuse A3's
stochastic guard on the stack metric too.

2. Does "goal met" require the *final* pass's cards to pass, or any pass historically? (Ratchet

3. Interaction of stack `onFail` (stop/continue/backoff) with goal-seeking — if a card fails
   but the goal could still be met next pass, does `onFail: stop` still halt? (Precedence
   rule needed.)
4. Where does the stack-level no-progress window K live — stack guardrails next to `onFail`?
