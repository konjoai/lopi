Guardrails vs Evals the settled two-axis model

Two questions, two axes, two buttons. Guardrails answer "should this loop keep running?" — the run envelope: budget, max-iterations, the no-progress window, on-fail policy, schedule. Evals answer "is the work actually good?" — the acceptance checks that decide whether an iteration counts as done. This is not a cosmetic rename: gate/until, previously grouped under "Limits" as pre/post-run shell commands, are reframed as the test tier of the eval ladder — a shell exit code is a verdict on quality, not a precondition on whether to run. See the flagged reconciliation below.

Guardrails

budget (auto/200k/none) · max-iterations (×N/∞) · window (no-progress halt) · on-fail (stop/continue/backoff) · schedule (cron, budget:time). Edited via the shield button → click-popover. Card shows one guardrails summary line at rest.

Evals

baseline → test → judge → suite, the eval ladder. Edited via the check button → click-popover, flat checklist layout (not grouped/suite-first — those were explored and rejected for this surface). Card shows one evals summary line at rest.

Two-button popover model (settled). Both guardrails and evals edit via a floating popover anchored to a card-bar button — shield for guardrails, check for evals — exactly like the existing schedule popover. The card itself never grows past its two summary lines regardless of how many evals are enabled; the popover floats above the stack so a dense multi-pane view never reflows. This superseded an earlier inline-slide-down exploration, which caused reflow in a multi-card stack, and an earlier suite-first/grouped-by-tier popover layout, which added a click before you could see or toggle an individual check — the flat checklist won because most edits are "flip one box," not "pick a bundle."

Autonomy — per-loop selector

First-class setting in the selector row, alongside model/effort/repo. An L1–L4 ladder controlling how much a loop may act without a human.

flagged, not resolved The mockup's L1–L4 labels don't match the backend enum they're meant to bind to. lopi-creation-flow.html's autonomy ladder uses a general "leash" framing — L1 writer · asks before acting, L2 director · acts, checks in, L3 advisor · runs, reports, L4 autonomous · only escalates. The real backend type, crates/lopi-core/src/loop_config.rs::AutonomyLevel, is narrower and PR-flow-specific: L1 ReportOnly (produce a report/diff artifact only, never open a PR), L2 DraftPr (open a draft PR, human approves — the default), L3 VerifiedPr (run the verifier before opening a PR), L4 AutoMerge (auto-merge when the verifier passes and the score clears the gate). These are not the same ladder — the mockup's L3 ("advisor, runs and reports") doesn't obviously map to the real L3 ("verified PR"), and the real ladder has no "asks before acting" step at all. Do not build the selector against the mockup's copy verbatim. Before UI-2 binds this control, either (a) relabel the UI to the real PR-flow semantics, or (b) treat this as a genuine product decision: extend/rename the backend enum to match the broader "how much leash" concept the mockup implies. Flagging per this doc's backend-binding-honesty rule rather than picking silently.

The one that's easy to miss: per-card event routing. The AgentEvent bus already exists and streams fine to the TUI, but true multi-card-per-pane output needs each event tagged with which card produced it. The unblocked fallback — one active card per pane, route by task_id — matches the mockups' actual behavior (only the "next" card ever runs), so this doesn't have to gate UI-4's start.
Output-attachment behavior (settled): collapsed by default to a 2-line strip so a running pane stays quiet; expand per-card to see detail; the container grows to fit and only thinking is height-capped with its own scroll, since it's the one stream that runs long. The live eval pill sits in the collapsed strip's header, giving the pass/fail/running rollup without expanding.
Overview is mostly free, except one column. The zoom-out surface is aggregation over the same AgentEvent stream and stack state the panes already use, so grouping (by state / by project), the stat bar, and drill-in are pure UI. The exception is the needs-you column: "waiting on a human" is derived from a verifier fail, a failing test-tier eval, or an :escalate severity, none of which the runner emits as a single signal today. That derivation, now including escalation severity, is the only real backend work the overview introduces.

PART 4 End-to-end test plan

Three tiers: every component in isolation, every setting's states, then the combinations. Full combinatorial coverage is thousands of cases and not worth it; the plan tests each dimension fully once, then the common and dangerous combinations exhaustively.

Tier B · Shorthand / grammar tests (the :action layer)

The same spec string must parse identically in composer, CLI, and Telegram. Pure-function tests, cheap, run them exhaustively.

  • Aliases: each of :research :plan :scope :goal :progress :implement :optimize :ratchet :benchmark :kill-test :tidy :handoff :kiban-sprint resolves to its expansion, including its default eval suite.
  • Modifiers: xN, x∞, at:"cron", every:15m, budget:auto, budget:200k, budget:time (wall-clock/count), max:N, @repo each parse to the right field.
  • Eval shorthand: a test-tier command modifier (successor to gate:"cmd"/until:"cmd") attaches an EvalDef, not a precondition — the parsed result must land under evals, not guardrails.
  • Templates: test {repo} until {cmd} fills; unfilled hole errors by name; {{ }} escapes.
  • Skill args: :kcqf vectro → skill + arg; :kcqf → empty arg; bad name fails loud.
  • Chains: :benchmark @vectro > :report @konjo.ai parses to two ordered steps.
  • Escalation: :escalate and an on-fail:escalate guardrail option each parse to a severity-tagged needs-you route.

Coverage strategy

  • Full once per dimension: every alias, every modifier, every selector option, every control button gets one isolated test (Tiers A + B). That's linear, cheap, and catches the "this one option is broken" class of bug.
  • Combinations by risk, not by cross-product: Tier C tests the ~12 stacks people actually build plus the dangerous ones (verifier fail, eval fail + escalate, ratchet revert, ∞ + budget). Full combinatorial is ~thousands of cases; this is ~12 that cover the real interaction surface.
  • Stubbed runner for determinism: Tier C runs against a fake runner returning scripted verdicts/tokens/exit codes, so tests are fast and deterministic. One thin "smoke" suite hits a real Claude call to prove the wiring, gated out of CI by default (cost).
  • The parsers get property tests: cron round-trip and template resolve are pure and total; fuzz them (arbitrary cron strings, arbitrary {holes}) since they're the load-bearing correctness surface and cheap to hammer.
  • Konjo gate: statistical rigor doesn't apply to UI tests, but the honesty rule does — a flaky test is quarantined and fixed, never retried-until-green. Negative-path tests (eval fails, verifier fails, budget defers, ratchet reverts) are first-class, not afterthoughts.
