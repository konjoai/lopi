The one that's easy to miss: per-card event routing. The AgentEvent bus already exists and streams fine to the TUI, but the output attachment needs each event tagged with which card produced it, so the right container fills. Without that tag, a multi-card stack can't show per-prompt output. Small change, but it gates the whole live-output surface, so land it early in the UI phase.
Output-attachment behavior (settled): collapsed by default to a 2-line strip so a running pane stays quiet; expand per-card to see detail; the container grows to fit and only thinking is height-capped with its own scroll, since it's the one stream that runs long. Two open knobs: (1) the strip shows the single most-recent activity line today — alternative is two fixed lines (current action + current tool) always both visible; (2) on expand, thinking defaults open — alternative is defaulting the most-recent completed section open. Both are one-line changes, decide during UI-4.
Overview is mostly free, except one column. The zoom-out surface is aggregation over the same AgentEvent stream and stack state the panes already use, so grouping (by state / by project), the stat bar, and drill-in are pure UI. The exception is the needs-you column: "waiting on a human" is not a signal the runner emits, it's derived from a verifier fail, a non-zero gate exit, or an escalation that doesn't exist yet. That derivation is the only real backend work the overview introduces, and it's the same escalation/triage primitive flagged in the feature research. The grouping is free; the most valuable column has a prerequisite.

PART 3 UI build sequence

The current web/ is a conventional dashboard (AgentGrid, AgentPane, AgentCard). The loop-stack is a full replacement of that surface. Sliced kill-test-gated, same discipline as the backend sprints, each slice shippable.

  • UI-0 · Recon. Read the real web/src/lib/components/* and routes; map what to keep (auth, layout shell, WS client) vs replace. Output UI_PLAN.md. No code.
  • UI-1 · Static stack. Prompt card, connector, next tag, the composer-at-top + model/effort/repo selector row. Renders from a real stack config, no interactivity yet.
  • UI-2 · Card controls. Loop pill + steppers, cron popover (two-way sync), Limits popover, verifier check button, duplicate/drag/delete/insert. Writes back to config.
  • UI-3 · Run + live controls. Run split (all four modes), the pause/drain/bump/kill row, running-outline state driven by the real event stream.
  • UI-4 · Output attachment. The container under the running card: collapsed 2-line activity strip by default, expandable to accordions + filter chips, thinking-only height cap, per-card event routing consumed from the WS stream.
  • UI-5 · Multi-pane. Side-by-side panes, hover-add-pane, per-pane run state, pane close.
  • UI-6 · Polish. Dark scrollbars, reduced-motion, empty states, keyboard, the orb-as-status.
  • UI-7 · Overview. The zoom-out: by-state board + by-project portfolio behind one toggle, global stat bar, needs-you filter, drill-in to pane. Pure aggregation over existing state, except the needs-you column, which needs the derivation backend first. Ships after the panes exist to drill into.

PART 4 End-to-end test plan

Three tiers: every component in isolation, every setting's states, then the combinations. Full combinatorial coverage is thousands of cases and not worth it; the plan tests each dimension fully once, then the common and dangerous combinations exhaustively.

Tier B · Shorthand / grammar tests (the :action layer)

The same spec string must parse identically in composer, CLI, and Telegram. Pure-function tests, cheap, run them exhaustively.

  • Aliases: each of :research :plan :scope :goal :progress :implement :optimize :benchmark :kill-test :tidy :handoff :kiban-sprint resolves to its expansion.
  • Modifiers: xN, x∞, at:"cron", every:15m, gate:"cmd", until:"cmd", max:N, budget:auto, @repo each parse to the right field.
  • Templates: test {repo} until {cmd} fills; unfilled hole errors by name; {{ }} escapes.
  • Skill args: :kcqf vectro → skill + arg; :kcqf → empty arg; bad name fails loud.
  • Chains: :benchmark @vectro > :report @konjo.ai parses to two ordered steps.
  • Guard is a shell exit code, not a DSL: gate:"exit 1" blocks, gate:"true" passes.

Coverage strategy

  • Full once per dimension: every alias, every modifier, every selector option, every control button gets one isolated test (Tiers A + B). That's linear, cheap, and catches the "this one option is broken" class of bug.
  • Combinations by risk, not by cross-product: Tier C tests the ~10 stacks people actually build plus the dangerous ones (verifier fail, gate fail + backoff, ∞ + budget). Full combinatorial is ~thousands of cases; this is ~10 that cover the real interaction surface.
  • Stubbed runner for determinism: Tier C runs against a fake runner returning scripted verdicts/tokens/exit codes, so tests are fast and deterministic. One thin "smoke" suite hits a real Claude call to prove the wiring, gated out of CI by default (cost).
  • The parsers get property tests: cron round-trip and template resolve are pure and total; fuzz them (arbitrary cron strings, arbitrary {holes}) since they're the load-bearing correctness surface and cheap to hammer.
  • Konjo gate: statistical rigor doesn't apply to UI tests, but the honesty rule does — a flaky test is quarantined and fixed, never retried-until-green. Negative-path tests (gate fails, verifier fails, budget defers) are first-class, not afterthoughts.
