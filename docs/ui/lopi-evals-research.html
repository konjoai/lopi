lopi · evals research

Why evals are hard, and how lopi makes them easy

The reasons evals are painful are well-documented and specific. The good news: most of them apply to a problem lopi doesn't have. The eval problem the whole industry struggles with is grading open-ended, subjective, non-deterministic output against ground truth that doesn't exist. lopi's loops mostly produce code and artifacts that have real, cheap, deterministic checks, tests, compilers, scanners, diffs. That difference is the entire opening. This maps each hard problem to whether it's lopi's problem, and derives the design that stays on the easy side of the line.

01 Why evals are structurally hard

Six reasons recur across every serious source (LangChain, the Pragmatic Engineer / Hamel Husain, DeepEval, the "hardest unsolved problem" essays). They fall into two buckets: theory-hard (intrinsic to grading LLM output) and tooling-hard (the setup friction that kills eval programs before they start). The second bucket is the one lopi can win outright.

Theory-hard (intrinsic to judging LLM output)

1. No ground truth
For most LLM tasks there's no single correct answer to compare against. "A response can be factually accurate but have the wrong tone, or sound persuasive while being completely wrong." Classical testing assumes a known-correct output; open-ended generation has none.
Mostly not lopi's problem. A coding loop's output is checked by cargo test, a compiler, a benchmark number, a scanner exit code, ground truth exists and is cheap. lopi only hits this wall for the subjective slice (is this prose good? is this design sound?), which is exactly where the judge-tier eval and human review come in. Design consequence: lean hard on deterministic evals; reserve the model-judge for the genuinely subjective.
2. Non-determinism (even at temperature 0)
The same prompt yields different outputs across runs. A single pass isn't representative; you have to think statistically (run N times, look at pass rate), which most tooling doesn't do by default.
lopi already solved this for itself. The Konjo standard is 30-run paired Wilcoxon, p<0.05, CoV gates. That statistical-rigor muscle is exactly what eval-under-non-determinism requires, and almost no eval tool bakes it in. Design consequence: an eval can specify "run N times, require pass-rate ≥ X," and the stats machinery already exists. This is a differentiator, not a gap.
3. The judge needs judging (LLM-as-judge reliability)
Using a model to grade output is the standard move (80–90% agreement with humans at ~500x lower cost), but the judge itself is unreliable: it must be validated against human judgment (TPR/TNR), it can be inconsistent, and it can't reliably grade nuanced quality. "We could verify the counts automatically but couldn't get reliable LLM evaluations of the quality."
Partly lopi's problem, and partly already handled. The "never grade your own homework" rule (verifier ≠ worker model) is the first reliability guard, most tools don't enforce it. What lopi should NOT pretend to solve: turning the model-judge into a rigorous, human-aligned scorer. Design consequence: present the judge as one signal among several, never the sole gate; always pair it with a deterministic eval where one exists; surface the judge's reasoning so the human can overrule.

Tooling-hard (the friction that actually kills eval programs)

4. Golden-dataset friction
The dominant eval paradigm requires a "golden dataset", a curated, versioned set of input/expected-output cases. It's "the single most important asset," it takes real work to build, and it rots: cases go stale, need weekly curation, and suffer data leakage that silently inflates scores. This is the #1 reason eval programs stall, most people never build the dataset.
This is the big one, and lopi sidesteps it. A golden dataset exists to grade a product's responses to user inputs. lopi isn't grading a chatbot against 500 curated questions; it's grading did this loop's change pass the repo's own tests / build / scan. The "dataset" is the repo's existing test suite, which already exists, is already versioned in git, and is maintained as a side effect of normal development. Design consequence: lopi's default evals need NO dataset. The test suite IS the eval set. That removes the single biggest source of friction for the coding-loop use case.
5. Two-tool sprawl + integration glue
"You almost certainly need two tools", a CI-gating framework (DeepEval/Promptfoo) plus an annotation/monitoring platform (LangSmith/Braintrust). They don't connect: "results aren't surfaced inline with traces," "traces and regression testing remain separate workflows," you manually wire failures back into tests. Setup means SDKs, env vars, YAML with cryptic __expected1/__expected2 columns, and self-host deployment headaches. "Workflow friction kills eval programs faster than any technical limitation."
This is the wedge, stated plainly. Every tool is a bolt-on: you instrument, you wire, you glue trace→eval→regression by hand. lopi's advantage is that the loop, the trace, the eval, and the gate are the same object. The eval isn't a separate platform you integrate; it's a property of the loop that already has the trace (the live output) and already controls the gate (continue/stop). Design consequence: evals are inline by construction, no second tool, no glue. The thing everyone lists as the killer is the thing lopi gets for free by owning the loop.
6. Evals saturate / get gamed
A fixed gate gets optimized against: "prompt-engineering-to-the-test", you tweak until this eval passes. Static benchmarks saturate (MMLU at 88%+). "When experiments run 100x faster than a human can manage, the eval pipeline becomes the constraint." Evals must evolve alongside the work.
Real risk, especially with :ratchet loops. An infinite optimize-loop WILL game a weak eval, that's the "framework becomes the project" trap in a new guise. Design consequence: the adversarial-reviewer eval is not optional garnish; it's the anti-gaming layer. And the honest-negative-results ethos (record what the eval missed) is the human-side defense. lopi should make it easy to add a case when an eval is found lacking, closing the loop the way Braintrust's "production failure → one-click regression test" does, but for your repo's own suite.
The one-sentence version: the industry's eval pain is 80% "grading subjective output with no ground truth, via bolt-on tools that need a hand-built dataset." lopi's loops mostly produce verifiable artifacts checked by tools that already exist in the repo, and lopi owns the loop, so the trace-eval-gate glue is free. lopi isn't solving the hardest version of the eval problem; it's serving the version where the answer is clean, and that version is huge and underserved.

02 The design this dictates

The research points at one architecture. Two separate axes, an eval ladder that's easy by default, and evals attached to presets so users never start from a blank slate.

Guardrails vs Evals, the two axes

ConceptQuestion it answersContainsOld name
Guardrails"Should this loop keep running?"budget, max-iterations, window, on-fail, schedule — the run envelopeguards / limits
Evals"Is the work actually good?"the acceptance checks: exit-code, tests, judge, suites — the quality bargate / until (partial)

Guardrails govern lifecycle; evals govern output. They compose (a failing eval triggers a guardrail's on-fail), but they're different categories, which is why "evals under guards" always felt wrong. Rename guards→Guardrails, promote Evals to a first-class peer.

The eval ladder, easy by default, powerful on demand

baseline
Execution succeeded. The agent's own actions didn't error (non-zero exit = fail). On by default, every loop, zero config. No dataset, no setup. This alone beats "spit code and hope." For a no-code research loop, baseline = "the loop's actions completed without error," not "a test passed."
test
A shell command's exit code is the verdict. cargo test, pytest, ./kill_test.sh. This is the existing gate/until, correctly reframed as an eval. Deterministic, cheap, no dataset, the repo's own suite is the eval set.
judge
A separate model grades the output against the goal/rubric. Enforces verifier ≠ worker. For the subjective slice where no deterministic check exists. Always paired with a deterministic eval where one is available; never the sole gate.
suite
A named, reusable bundle of evals. e.g. unit + integration + e2e tests + code review + vuln scan + adversarial reviewer. Presets attach these. KCQF is the canonical built-in suite — your CI Three-Wall gate, generalized into a user-facing primitive.
The "baked in everywhere" principle: every loop has the baseline eval whether or not the user configures anything. Adding a test eval is one field. Presets pre-attach suites. The result: a user never faces a blank eval config, and never ships a loop with zero acceptance checking. That default-on posture is what no other tool does, they all make evals opt-in, which is why 52% of teams skip them.

Presets carry evals (the color-coded stack types you wanted)

Choosing a preset sets the prompt and attaches its default eval suite, because different work needs different validation:

  • :research → baseline + judge (is the synthesis grounded / non-hallucinated?). No test suite; nothing compiles.
  • :implement → baseline + unit + integration + e2e tests + code review (judge) + vuln scan + adversarial reviewer. The full KCQF.
  • :optimize / :ratchet → baseline + benchmark-beats-best + the statistical gate (30-run, p<0.05) + adversarial reviewer (anti-gaming).
  • :benchmark → baseline + the measurement harness + CoV/variance gate.
  • :tidy → baseline + tests-still-pass + no-new-lint.

These are the "different types of prompt stack you choose from", and they're where color/glyph coding earns its place: not coloring by type on the board (state wins there), but a labeled, glyphed preset picker so the user reaches for "implement" and gets the right evals without thinking about it.

03 The enterprise extension further goal

Your instinct, plug evals into existing pipelines and third-party validators, is exactly where the market is moving, and it's a clean extension of the same model. Flagged as a later/enterprise goal, not near-term, but the architecture should not preclude it.

  • KCQF expands by language. The suite generalizes beyond Rust/Python to Java, C#, PHP, Ruby, each language's suite is just a different set of test/scan commands behind the same eval interface. The interface doesn't change; the commands do.
  • Third-party evals as first-class eval types. SonarQube, OWASP Dependency-Check, Trivy, Snyk, these are just evals whose "exit code" comes from an external service. They slot into the same ladder as the suite tier. The eval abstraction (a check that returns pass/fail + detail) already fits them.
  • Set-up-once, available-to-all. The admin-panel model: an org configures a SonarQube connection once; that eval type becomes selectable for every user's loops. This is the enterprise version of "presets carry evals", org-level eval suites. It's the LangSmith/Braintrust enterprise story, but eval-first and loop-native instead of a separate observability platform.
  • Why this is a real market: Gartner projects LLM-observability spend hitting 50% of GenAI deployments by 2028. Every competitor is an observability tool that bolted on evals; the field's own reviewers say the gap is "evals-first, built for whole teams, that block bad releases," not another tracing dashboard. An eval-native loop cockpit that gates on your existing validators is a differentiated enterprise wedge.
Scope discipline (Konjo rule): the enterprise vision is compelling and should shape the abstraction (an eval is "a check returning pass/fail + detail," source-agnostic) so third-party integrations slot in later without a rewrite. But building the SonarQube/Trivy connectors, admin panel, and org-eval-sharing now would be the framework becoming the project. Ship the local eval ladder first (baseline→test→judge→suite over the repo's own tools), prove it's clean and easy, then extend to external validators when there's a user asking for it. The architecture allows it; the roadmap defers it.

04 What goes into the plan

  • Rename guards → Guardrails (run envelope) across the plan; Evals becomes a new first-class concept, peer to Guardrails. now
  • Eval ladder: baseline (on by default) → test → judge → suite, with the baseline defined as "execution succeeded" so it applies to code and non-code loops. now
  • Presets carry eval suites; KCQF is the canonical suite; presets are the glyphed stack-types the user picks. now
  • Evals as a UI surface: an evals row on the card, pass/fail per eval when running, the judge's gaps/fix-hints surfaced (the comprehension-debt fix). now
  • Statistical evals: an eval can specify run-N-times / pass-rate / the existing 30-run gate, reusing the Konjo stats machinery. now
  • Eval abstraction is source-agnostic (pass/fail + detail), so external validators fit later without rework. design-for, build-later
  • Enterprise: KCQF multi-language, SonarQube/Trivy/Dependency-Check as eval types, admin-panel set-up-once. enterprise goal
