lopi · loop framework

A loop is a prompt
that runs again.

With a contract for what it reads and writes, a trigger for when, and guards for when to stop. Four primitives, two operators. Nothing else.

02 · the contract

Why ":progress" isn't vibes

General prompts work because every iteration runs inside the same read → run → write cycle. The write half is enforced: an iteration that records nothing is a failed iteration.

read · before
  • PLAN.md intended work
  • PROGRESS.md what's done
  • NEXT.md last handoff
  • logs + git live state
→
run · the prompt
  • one iteration
  • one Task in the queue
  • git-isolated branch
  • kiban discipline applies
→
write · after
  • PROGRESS.md append entry
  • NEXT.md rewrite handoff
  • ledger outcome + cost
⟲ next iteration reads what this one wrote
Negative results are written the same as wins. Tomorrow's :progress knows the speculative path was tried and reverted, instead of rediscovering it.
03 · guards

No unguarded infinity

x∞ is legal and useful, but max, budget, and window still apply unless you explicitly lift them. Implicit guards render dim on every card, so nothing invisible governs execution.

guardmeaningdefault
xN / x∞iteration countx1
until:"cmd"stop when the command exits 0, checked after each iteration—
gate:"cmd"pre-flight: must exit 0 before iteration 1, else the loop refuses to start—
max:Nhard ceiling regardless of until25
budget:token guard: auto self-meters the Claude 5h window and weekly cap from the ledger, keeps a reserve for interactive use, and parks loops in waiting·budget instead of failing · explicit N ceilings stack with auto · none opts outauto
pace:evenspread iterations across the window instead of back-to-back—
window:Twall-clock ceiling8h
on-fail:stop · continue · backoffstop
Conditions are shell exit codes. until:"cargo test" loops until that command passes. No condition DSL to learn: anything you can script, you can gate on.
04 · composition

Two operators. Wrapping does the rest.

A loop can wrap a single prompt or a whole chain. That one rule produces sprints, nightly pipelines, and scheduled batches without any new object to learn.

Stack · independent, in queue order
The queue you already have. Bottom card runs next.
:progress x3 , then :tidy , then "draft HN post"
Chain · output feeds the next step
Each step's writes + handoff become the next step's reads. One unit of scheduling.
(:plan > :implement > :test) x2
Aliases go all the way down: a saved chain or configured loop is just a bigger alias. :nightly can name :tidy > :kill-test at:"0 2 * * *". One concept, every scale.
