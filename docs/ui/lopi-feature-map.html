What to build after the core ships

The interface and the five primitives are settled. This is a prioritized map of additional features, each checked against what practitioners in loop engineering are actually doing and against the rasen contract. Nothing here changes the core: a stack stays a sequential column, a pane stays the unit of parallelism.

The field agrees with your design. The loop-engineering discipline named by Steinberger ("design the loops that prompt your agents"), Cherny ("my job is to write loops"), and Osmani converged on the same anatomy you already have: trigger, scope, action, budget, stop, report, plus a state file as the spine. Your primitives (prompt, contract, trigger, guards/limits) and your alias pack are that anatomy. So this list is not about filling gaps in the core. It is about the layer above it: reuse, verification, and reporting, which is where the field says the real leverage and the real danger both live.

Each of these is something loop-engineering practice treats as essential, and each extends a primitive you already have rather than adding a new axis.

Prompt templatesyour saved-prompt idea An alias with named holes: :bench {repo} until:"{cmd}". Picking it prompts for the blanks, or you fill them inline. A saved chain is just a bigger alias. Steinberger's durable half: "if you do something more than once, turn it into a skill." Claude Code skills already take $ARGUMENTS / placeholders. Verifier / checker stepmaker-checker split A loop can mark a step as verify: run by a second model that grades "done" against the goal, separate from the worker. Fails the gate if the check fails. Named the single most consequential design choice in a loop by every source. /goal uses a separate grader so the worker "does not grade its own homework." You have gate/until; this makes the grader a first-class, model-backed check. extends limits M P1 · now
Report / notify on finishREPORT primitive A stack can end by posting a summary somewhere: Telegram (you already have the gateway), Slack, a PR comment, a file. report:telegram as a stack-level setting. The sixth primitive in the practitioner anatomy: "let the findings come to you instead of going around checking." A loop without a report is a loop you have to babysit, which defeats the point. new primitive S P1 · now
No-progress detectionauto-stop Halt when the loop stops moving: same error N times, empty diff, identical output twice. A built-in implicit guard, on by default. Huntley/ralph practice: "stop when the same error appears N times in a row." Your ∞ default is only safe with this. It is the difference between a loop that halts and one that burns tokens in a wall. extends guards M P1 · now
Live cost / token meterbudget made visible Per-stack running tally of tokens and spend this run, next to the budget limit. The orb or footer shows burn rate; budget:auto already meters it, so surface it. Every source flags the same trap: token cost compounds ~10x faster than expected, and both Cherny and Steinberger work where budget is not a constraint. For a solo M3 builder it is the constraint. Seeing burn is how you trust ∞. surfaces budget:auto M P1 · now

If you build only this tier, lopi covers the full practitioner loop anatomy with the reuse and safety pieces that the field says separate a real loop from "a task, repeated."

If I had to pick the short list

Three that would move lopi from "a clean loop UI" to "the loop tool people reach for," in order:

1. Prompt templates. It is your own instinct and it is the most-cited reuse pattern in the field. It also costs you almost nothing because an alias with holes is a tiny grammar extension, and it makes every other feature more reusable.

2. A verifier step. The field is unanimous that the maker-checker split is what makes an unattended loop trustworthy. You already concede TTFT and cherry-picked variance honestly; a built-in "don't grade your own homework" check is the same value in the product, and it is exactly the kind of correctness story that reads well to the labs you are applying to.

3. Report-on-finish over Telegram. You already have the gateway. Closing the loop with "the findings come to you" is what turns a stack from a thing you watch into a thing you leave running, which is the entire premise Steinberger and Cherny are pointing at.
